## Overview

A Background function is the unit of work introduced to define a piece of code
that will execute in the background on the [Defer Platform](/platform/overview).

A Background functions can be defined as a [CRON](/features/cron),
[executed as delayed](/features/delays), and composed to build a
[Workflow](/features/workflows).

## Conventions

A Background function is:

1. a **named `async`** function
1. with **serializable arguments**
1. as **the default export** of
1. a file located in a **`defer/` folder**

### Default export

<CodeGroup>

```ts Correct
import { defer } from "@defer/client";

const weeklyBrief = async () => {
  // ...
};

export default defer.cron(weeklyBrief, "5 0 * * *");
```

```ts Incorrect
import { defer } from '@defer/client'

async function weeklyBriefFn () => {
  // ...
}

// Problem: Named export instead of a default export
export const weeklyBrief = defer.cron(
  weeklyBriefFn,
  "5 0 * * *"
)
```

</CodeGroup>

### Async function

<CodeGroup>
```ts Correct
import { defer } from "@defer/client";

const weeklyBrief = async () => { // ... };

export default defer.cron(weeklyBrief, "5 0 * * *");
```

```ts Incorrect
import { defer } from '@defer/client'

function weeklyBrief () => {
// ...
}

// Problem: `weeklyBrief()` is not `async`
export default defer.cron(
  weeklyBrief,
  "5 0 * * *"
)
```

</CodeGroup>

### Named function

<CodeGroup>

```ts Correct
import { defer } from "@defer/client";

const weeklyBrief = async () => {
  // ...
};

export default defer.cron(weeklyBrief, "5 0 * * *");
```

```ts Incorrect
import { defer } from "@defer/client";

// Problem: The wrapped function is anonymous
export default defer.cron(async function () {
  // ...
}, "5 0 * * *");
```

</CodeGroup>

<br />

<Warning>
**A Background function's name should be unique in a project**

The Background function's name is used as a unique identifier when triggering
and listing executions.

</Warning>

### Serializable arguments

<CodeGroup>
```tsx Correct
import { defer } from "@defer.run/client";

async function importContacts(userId: string) { // ... query information using
`userId`... }

export default defer(importContacts, {
concurrency: 10,
retry: 5,
});
```

```tsx Incorrect
import { defer } from "@defer.run/client";

async function importContacts(prismaClient: PrismaClient) {
  // ...use the `prismaClient` instance...
}

// Problem: The `prismaClient` argument is not serializable
export default defer(importContacts, {
  concurrency: 10,
  retry: 5,
});
```

</CodeGroup>

<br />

<Info>
**Why should arguments be serializable?**

As covered in ["How Defer works?"](/platform/overview), a call to a Background
function triggers a call to the Defer Platform API with the environment token,
function name, and arguments.

Hence the necessity to have serializable arguments.

</Info>

## Motivation

The concept of the Background function brings seamless integration with all
codebases, allowing you to _move existing parts of your code in the background_
without refactoring and writing new use cases without restrictions.

Contrary to Lambdas and Edge Functions, the Background function can leverage all
installed dependencies (natives included) and import local files, without any
build configuration.

The following example works out of the box:

```tsx defer/importContacts.ts
import { defer } from "@defer/client";
// support for native dependencies
import { hash } from "bcrypt";
// support for custom paths for local imports
import { SALT_ROUNDS } from "~/utils/crypto";

const bcryptPassword = async () => {
  const myPlaintextPassword = "s0//P4$$w0rD";
  hash(myPlaintextPassword, SALT, function (err: any, hash: any) {
    console.log(hash);
  });
};

export default defer(bcryptPassword);
```

The unified experience around Background functions unlocks easy access to
execution configuration ([retries](/features/retries),
[concurrency](/features/concurrency)) and
[workflow development patterns](/features/workflows).
